# Stage 3 plan

Wrote `plan/STAGE3.md` per the user's explicit three-goal request:

1. A 3D rendering library that can visualize the trajectory.
2. A visualization application showing that 3D view next to the
   corresponding video frames and diagnostic graphs.
3. Per-run browsing — users can pick which run's results to view, not
   just the most recent one.

New crates/binaries planned: `crates/slam-render` (hand-written camera/
scene/primitive code on `wgpu`, this stage's actual "rendering library"
deliverable — see `decisions/0018` for why it's hand-written while
`egui`/`egui_plot` UI chrome and graph widgets are treated as infra, the
same "own the algorithm, buy the infra" split as Stages 1-2) and
`bin/slam-viz` (the application). Eight milestones (M0-M7): M0 is a
small `bin/slam-run` backend prerequisite (multi-run, non-clobbering
output layout — needed before goal 3 can mean anything, since today's
`bin/slam-run` overwrites `runs/<sequence>/` on every invocation), M1-M2
build `slam-render`, M3-M6 build `bin/slam-viz`'s three synced panels,
M7 is goal 3's actual delivery (a real per-run picker with enough
metadata to choose meaningfully).

Scoped as post-hoc visualization of completed `bin/slam-run` output,
explicitly not live/in-loop visualization during a run — threading
visualization into the real-time-critical VIO loop would risk the
real-time bar Stage 2 just spent two milestones earning back
(`memory/progress/2026-07-21-stage2-m5-real-time-bar-met.md`).

Also updated `CLAUDE.md`'s "current stage plan" pointer from
`plan/STAGE2.md` to `plan/STAGE3.md` (Stage 2 is done — both its goals
were met, see `2026-07-22-stage2-m6-huber-and-window-size-tried-
reverted.md`) and added a plan-level dependency-policy addendum since
this is the first stage to touch graphics/UI, a domain the original
Stage 1 "infra crates fine, algorithms hand-written" policy
(`decisions/0001`) didn't have to define precisely until now.

One open methodological question this plan had to answer itself, not
just defer: CLAUDE.md's verification rule (`cargo test` + a plain-text/
CSV human-readable test app) doesn't translate literally to a GUI
deliverable. `plan/STAGE3.md` has a "Verifying a GUI deliverable"
section addressing this directly — push everything testable (camera
math, data adapters, time-sync logic) into real `cargo test` coverage,
and treat "run `bin/slam-viz` and look at it" as this stage's
equivalent of reading `bin/slam-inspect`'s output, rather than trying to
force a plain-text substitute for an inherently visual deliverable.
