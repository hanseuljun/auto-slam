# Stage 4 plan

Wrote `plan/STAGE4.md` per the user's explicit three-goal request,
right after Stage 3 closed out:

1. `bin/slam-run` runs the whole sequence by default, not the 600-frame
   (~30s) bounded clip Stage 2's M0 introduced.
2. The real-time bar (factor ≤ 1.0) must hold on the full sequence.
3. Accuracy must not regress relative to the current bounded-clip
   numbers already in `docs/RESULTS.md`.

## This gap was predicted, not discovered

Two things already on record said this exact gap existed and was
unclosed: `plan/STAGE2.md`'s own Risks section ("M0's default-run
truncation could hide the exact scaling problem this stage exists to
fix... a truncated clip that happens to fit inside the window can look
real-time for reasons that have nothing to do with actually fixing the
scaling") and `docs/RESULTS.md`'s own limitations section ("not
re-benchmarked with `--full` yet"). Stage 2 asserted the M1
marginalization fix should generalize to full sequences but never
measured it — Stage 4 exists to actually check.

## A concrete, code-grounded root-cause lead (not a guess)

Read `slam-backend/src/vio.rs`'s `global_bundle_adjustment_inner`
before writing the plan: it builds its `Problem` from
`self.history.iter().chain(self.window.iter())` — every keyframe ever
created, not the bounded window Stage 2 M1's marginalization actually
bounds. `slam-optim/src/solver.rs`'s dense O(dim³) solve (the exact
scaling `plan/STAGE2.md`'s original M9-rollback root cause named) was
never replaced for this call site — Stage 2's M3 deferred a sparse
solve, but that deferral's own reasoning ("fine at M1's now-bounded
problem sizes, window ~8 keyframes") only covers the *windowed*
solver, not `global_bundle_adjustment`, which still solves over the
entire history. A full sequence has ~3-6x more keyframes than the
600-frame bounded clip (per `bin/slam-run`'s own "bounded run: 600/N
frames" output across the five sequences: 3682/3040/2700/2032/2273) —
an O(n³) solve over 3-6x more keyframes is a ~30-200x slowdown for
global BA specifically, independent of whether the per-frame VIO loop
(already confirmed real-time in Stage 2 M5) has any problem at all.
This is the leading hypothesis `plan/STAGE4.md`'s M0/M1 should confirm
or rule out with real profiling — written into the plan as a lead, not
a diagnosis, same "measured, not assumed" discipline every prior stage
held to.

## Real, if inconclusive, evidence this session

Attempted `slam-run --full data/machine_hall/MH_01_easy` twice this
session (once from this agent, once apparently from the user in a
separate terminal running `--full` over all five sequences) — both
processes ran for many minutes (the agent's own attempt: 10+ minutes,
observed via `ps` CPU-time growth) and then disappeared without
writing any new `runs/MH_01_easy/<run_id>/` entry or updating the
latest-snapshot `trajectory.csv`. Inconclusive on its own (long-running
background Bash processes proved unreliable to wait on this session —
worth noting as a process/tooling risk, not proof of an algorithmic
hang), but consistent with, not contradicting, the predicted gap above.
Recorded in `plan/STAGE4.md`'s "What we already know" and its own
Risks section (M0 should run in the foreground with an explicit
ceiling, not rely on an unattended background process surviving many
minutes).
