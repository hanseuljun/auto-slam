# Stage 2 plan

## What happened

- Attempted Stage 1's M9 (evaluation harness & benchmarking) directly:
  built RPE + CSV reporting into `slam-eval` and a `bin/slam-run` crate
  that runs the full M5+M8 pipeline (windowed VIO + one global bundle-
  adjustment pass) over a complete, un-truncated sequence. Ran it on
  `MH_01_easy` (~184s of data) — it did not finish in under ~30 minutes
  wall-clock. All of that work was uncommitted and got rolled back
  (`git checkout`/`rm` back to the last pushed M8 commit) rather than
  landed half-verified.
- Root-caused the slowness by reading code, not guessing: `slam-optim`'s
  `build_normal_equations`/`optimize` (`crates/slam-optim/src/solver.rs`)
  use a **dense** `DMatrix` normal-equations system and a dense LU solve,
  sized by `(problem.keyframes.len() - 1) * 15`. M5's sliding window never
  marginalizes the oldest keyframe (`decisions/0007`) — it either drops it
  (windowed path) or, as of M8, retains it forever in `history`. So
  `global_bundle_adjustment`'s problem size — and its O(dim³) per-
  iteration solve cost — grows with the *entire trajectory length*, not
  a bounded window. This fully explains the observed wall-clock, is
  independently verifiable by reading the cited file, and doesn't depend
  on the rolled-back code to be true (the dense-solve architecture is
  unchanged, current code).
- Wrote `plan/STAGE2.md`: a 7-milestone (M0-M6) plan with two goals —
  (1) real-time VIO (processing N seconds of data through the continuous
  tracking + windowed-backend loop takes <= N seconds wall-clock, real-time
  factor <= 1.0) and (2) finishing Stage 1's two never-landed milestones,
  M9 and M10. Ordered so M9 (with timing instrumentation added) lands
  first to get a measured baseline, then marginalization (closes
  `decisions/0007`) + analytic IMU Jacobians (closes `decisions/0006`) +
  a sparse-aware solve + `rayon` parallelism address the real-time goal,
  then M10's accuracy tuning lands last (deliberately, so tuning iteration
  is fast by the time it starts) — see the plan file itself for the full
  rationale and per-milestone detail.

## State at end of session

`plan/STAGE2.md` exists; no Stage 2 code has been written yet (the one
real attempt at M9-shaped work this session was fully rolled back — repo
is exactly at the M8 commit, `crates/slam-eval` and `bin/` unchanged from
that state). Next session should start at Stage 2's M0: rebuild the
evaluation harness, this time with per-stage wall-clock timing built in
from the start (not bolted on after), and a bounded/truncated default run
in `bin/slam-run` so the harness itself stays practical to iterate with
while the rest of the stage is in progress.

## Lesson for next time

Don't build a full-sequence benchmarking tool without first checking
whether the thing being benchmarked has a known-unbounded cost. The
"unbounded `history`" architecture (decisions/0007) was already documented
as a known gap before this session's M9 attempt — the mistake was running
the expensive full-sequence case *before* establishing a fast baseline or
truncated mode to iterate with, so the first sign of trouble was a
30-minute wait rather than a quick timing number. Stage 2's M0 explicitly
builds timing instrumentation and a bounded default run mode before
anything else, specifically to avoid repeating this.
