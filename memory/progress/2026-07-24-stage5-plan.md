# Stage 5 plan

Wrote `plan/STAGE5.md` per the user's two-goal request, right after
Stage 4 closed out:

1. ATE near the start of a trajectory should be near zero — ground
   truth is available from t=0, so there's no time to have drifted yet.
2. Sequences that return to their starting point should show clear
   evidence of proper loop closure; right now ATE stays high regardless.

Didn't just write these down — spent real investigation time confirming
both with actual numbers before drafting the plan, matching this
project's own "measure before fixing" habit even at the planning stage.

## Goal 1 confirmed, and its mechanism identified

Compared `MH_01_easy`'s raw (pre-alignment) pose estimates for the first
~20 keyframes between the bounded 600-frame run and the full 3682-frame
run: nearly identical (diffs ~0.03-0.10m). But the *reported* ATE at
that same point (after `crates/slam-eval/src/align.rs`'s Umeyama
alignment) is 0.18m on the bounded run and 3.1m on the full run — 17x
different for frames whose underlying estimate barely changed. Root
cause: Umeyama fits a single similarity transform over *every* point in
the trajectory at once; once later keyframes have drifted far (confirmed
real by `plan/STAGE4.md` M2), the least-squares fit trades off matching
those against matching the still-accurate early portion, spreading later
drift backward onto frames that were never actually wrong. Verified the
mechanism directly: re-aligning using only the first 10 keyframes drops
err[0] to 0.07m, but blows up the rest of the trajectory to 88m rmse
(138m at the end) — proving the *effect* (Umeyama masks the early error)
while also showing the naive fix (tiny anchor window) is itself unstable
(a lever-arm effect from a poorly-conditioned small window's rotation
uncertainty). `plan/STAGE5.md` M0/M1 exist to find a principled fix, not
assume the first thing that worked in a spot-check.

## Goal 2 confirmed at dataset scale, not just MH_05

`memory/notes/dataset-quirks.md` already flagged MH_05's loop. Checked
all five sequences' raw groundtruth this session (closest-approach pair
requiring >=15s separation *and* real distance traveled in between, to
rule out "just hovering"): **all five** return within ~0.13-0.26m of
their start after 70-130m of real flight. This is the dataset's own
structure (`machine_hall` sequences take off and land near the same
spot), not a one-sequence special case. Also confirmed `crates/
slam-loopclosure` (Stage 1 M7) is real, tested code that's simply never
wired into `bin/slam-run` (`loop_closure_seconds: 0.0` hardcoded) — the
only place it runs at all is `bin/slam-inspect`'s demo, hardcoded to
`MH_05_difficult` only, on `VoPipeline` (no IMU/backend/BA). Re-ran that
demo fresh: `ATE without=5.613m with=3.293m` — real improvement, but far
from "properly closed" for a loop that's actually 0.13m tight in ground
truth, and never measured on the other 4 sequences or the real pipeline.

## An unprompted third finding, folded into goal 1's scope

While investigating goal 1, found the alignment's own fitted *scale*
parameter is far from 1.0 (0.017-0.47 depending on window/sequence,
worse the longer a run continues) — surprising for a stereo-inertial
system where scale should be directly observable and near-1.0 if
reconstruction is metrically sound. Directly measured: raw estimated
path length vs. ground truth's own path length is off by 5-9x (est
travels 715.75 "m" vs. gt's real 78.79m on the full `MH_01_easy` run),
worse than the bounded clip's own 5x ratio — consistent with compounding
drift (possibly connected to `plan/STAGE4.md` M2's 45-52% track-loss
recovery rate, a hypothesis not yet confirmed) rather than a fixed
calibration constant. Didn't invent a third headline goal for this —
folded it into `plan/STAGE5.md` M0 as something the metric-honesty work
must actually root-cause, since a free-scale (Sim3) alignment is exactly
the kind of thing that can hide a problem like this, and published
stereo-inertial SOTA numbers conventionally use fixed-scale alignment
(this repo's existing SOTA comparison table may not be quite
apples-to-apples on this axis either).
